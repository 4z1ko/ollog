## Milestone Summary

**Key Accomplishments:**
1. Idempotent `normalize_time_on()` startup migration pads all 4-digit `TIME_ON` records to `HHMM00` via anchored regex + aggregation pipeline — zero-op on re-run, 5 integration/unit tests
2. `parse_adif_datetime()` confirmed via explicit test coverage to accept both HHMM (4-digit) and HHMMSS (6-digit) — no code change needed (existing function)
3. QSO_DATE and TIME_ON wrapped in Heroicons padlock buttons — `readonly` (not `disabled`) so values always reach POST body; locked styling renders before JS loads
4. Live UTC clock via `setInterval` using `getUTC*` exclusively — zero local-timezone leakage, ticks every second while locked, stops on unlock
5. HHMM→HHMM00 normalization in `htmx:beforeRequest` before validation fires; range-checking regex `/^([01]\d|2[0-3])([0-5]\d)([0-5]\d)$/` rejects out-of-range values
6. `localStorage`-backed reset-mode toggle ("Reset to live UTC" / "Keep current date/time") persists across page reloads; all 14 requirements browser-verified in a live session

**Key Decisions:**
- `readonly` (not `disabled`) on locked fields — `disabled` silently drops field value from POST body
- Live clock uses `Date.getUTC*()` exclusively — never `getHours()`/`getDate()` which return local timezone
- HHMM normalization fires in `htmx:beforeRequest` (already present in form.html) before validate()
- Post-submit behavior toggle reads/writes `localStorage`
- `hx-target="#qso-result"` points at a sibling div — form DOM, event listeners, and setInterval survive every submit; no re-initialization hook needed
- `parse_adif_datetime()` in service.py already handles both HHMM and HHMMSS — DB-02 is a server-side validation confirmation, not a code change
- Aggregation pipeline `[{$set: {TIME_ON: {$concat: ["$TIME_ON", "00"]}}}]` used server-side — no Python loop
- Anchored regex `^\d{4}$` in filter prevents double-padding on repeated migration runs
