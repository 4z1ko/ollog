<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser form POST -> profile_update() | Untrusted form data crosses into the server; user can submit arbitrary field values |
| profile_update() -> MongoDB $set | Validated data is written to the user's own document |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-45-01 | Tampering | profile_update() notify_sound field | mitigate | `(notify_sound == "true")` explicit string-to-bool conversion rejects any value other than `"true"` as `False`; `ProfileUpdateRequest(**raw)` with Pydantic validates the `bool` type before `update_profile()` writes to MongoDB |
| T-45-02 | Tampering | Operator A modifying Operator B's notify_sound | mitigate | `update_profile(user, updates)` receives `user` from `get_current_user_cookie` dependency -- the update target is always the authenticated user's own document (existing operator isolation pattern) |
| T-45-03 | Tampering | Extra form fields injected via hidden inputs | mitigate | `ProfileUpdateRequest(**raw)` Pydantic schema only accepts declared fields; extra fields raise ValidationError (existing pattern -- no change) |
| T-45-04 | Information Disclosure | notify_sound value leaked to other operators | accept | `notify_sound` is a non-sensitive boolean preference (on/off for sound). No PII. REST API GET /api/profile/ returns only the authenticated user's own profile (JWT-scoped). Low risk. |
</threat_model>
