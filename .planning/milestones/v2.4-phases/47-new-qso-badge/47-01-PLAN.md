(b) **Helper functions** -- After the `playTone` function (after line 156), add:
```javascript
    function updateBadge() {
      badgeText.textContent = newQsoCount === 1
        ? '1 new QSO'
        : newQsoCount + ' new QSOs';
      badge.classList.remove('hidden');
      badge.classList.add('flex');

    function dismissBadge() {
      newQsoCount = 0;
      badge.classList.add('hidden');
      badge.classList.remove('flex');
    }

    badge.addEventListener('click', dismissBadge);
```

CRITICAL pattern: `classList.remove('hidden')` BEFORE `classList.add('flex')` in updateBadge.
Tailwind's `hidden` compiles to `display: none !important` -- adding `flex` without removing
`hidden` first leaves the badge invisible. Reverse order in dismissBadge (add `hidden`, remove
`flex`). This mirrors the exact LIVE indicator pattern on lines 168-173.

(c) **Badge increment in htmx:sseMessage handler** -- Insert AFTER the sound notification
block (after line 179: `playTone(audioCtx);` closing brace) and BEFORE the existing
`if (!document.getElementById('auto-refresh-ok')) return;` on line 180.

Insert:
```javascript
      // Badge increment -- fires when page 2+ or filters active (D-04, D-06)
      if (!document.getElementById('auto-refresh-ok')) {
        newQsoCount++;
        updateBadge();
      }
```

CRITICAL placement: This block MUST go BEFORE line 180's
`if (!document.getElementById('auto-refresh-ok')) return;` -- otherwise it is unreachable.
Both conditions check for sentinel absence: the increment block increments and falls through;
the return guard on line 180 then stops execution before `htmx.ajax()`. When sentinel IS
present (page 1, no filters), the increment block does not fire, and execution falls through
to `htmx.ajax()` for auto-refresh. This is correct.

(d) **htmx:afterSettle auto-dismiss listener** -- Add as a new listener AFTER the
`htmx:sseClose` handler (after line 200), still inside the IIFE:


No target guard needed per RESEARCH.md Pattern 3 -- if `auto-refresh-ok` is present after ANY
settle event, the operator is on page 1 with no filters and dismissing is always correct (D-08).

| T-47-01 | Spoofing | SSE new_qso event | accept | SSE endpoint (/feed/station) is already authenticated via cookie; badge only increments a local counter -- no data exposure. Attacker would need valid session to reach SSE endpoint. |

| T-47-03 | Denial of Service | Rapid SSE events overflow counter | accept | Counter is a simple integer increment in JS; no DOM creation per event. Even 10,000 events produce only textContent updates. No memory or performance concern. |
