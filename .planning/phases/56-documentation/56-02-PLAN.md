    Confirm the two `!!! danger` blocks rendered as styled HTML admonitions, not as raw
    paragraphs containing the literal text `!!! danger`. This guards against the silent
    failure mode documented in RESEARCH.md Pitfall 1 (where `mkdocs build` succeeds but
    the admonition is treated as plain text because `markdown_extensions: [admonition]` is
    missing or misconfigured).

    Run all four checks from the repository root. All four must succeed:

    ```bash
    # 1. Operator page contains the rendered admonition HTML
    grep -l 'class="admonition danger"' site/operator-guide/profile/index.html

    # 2. Admin page contains the rendered admonition HTML
    grep -l 'class="admonition danger"' site/admin-guide/account-management/index.html

    # 3. Operator page does NOT contain the unrendered raw syntax (anchored to bol of an HTML paragraph or as literal text)
    ! grep -E '&gt;!!! danger' site/operator-guide/profile/index.html

    # 4. Admin page does NOT contain the unrendered raw syntax
    ! grep -E '&gt;!!! danger' site/admin-guide/account-management/index.html
    ```

    Also confirm both pages contain the new section heading IDs (MkDocs auto-generates
    them from the markdown headers):

    ```bash
    grep -q 'id="danger-zone"' site/operator-guide/profile/index.html
    grep -q 'id="clear-operator-log"' site/admin-guide/account-management/index.html
    ```

    If any check fails:
    - Missing `class="admonition danger"` → admonition extension not applied; check
      `mkdocs.yml` and rerun Plan 01 Task 1, then this plan.
    - Raw `!!! danger` text leaked into output → same root cause.
    - Missing section heading IDs → the markdown sections were not authored or were
      placed inside a code block; check Plan 01 Tasks 2 and 3.

    DO NOT modify any file under `site/` to "fix" verification failures — fix the source
    in `docs/` or `mkdocs.yml` and rebuild.
