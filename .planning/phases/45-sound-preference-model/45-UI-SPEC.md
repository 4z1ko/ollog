## Element Specifications

### Sound Notifications Form Section

Placement: inside the existing `<div class="grid grid-cols-1 sm:grid-cols-2 gap-5">` in the "Operator Details" card, after the TX Power field, before the closing `</div>`.

```
<div class="sm:col-span-2">
  <label class="form-label">Sound Notifications</label>
  <div class="flex items-center gap-3 mt-1">
    <input type="checkbox"
           name="notify_sound"
           value="true"
           {{ 'checked' if profile.notify_sound else '' }}
           class="w-4 h-4 rounded border-gray-300 dark:border-gray-600
                  text-indigo-600 focus:ring-indigo-500
                  dark:bg-gray-800 dark:checked:bg-indigo-600">
    <input type="hidden" name="notify_sound" value="false">
    <span class="text-sm text-gray-700 dark:text-gray-300">
      Play a tone when a new QSO arrives
    </span>
  </div>
  <p class="form-hint">Requires at least one page interaction (browser autoplay policy)</p>
</div>
```

Key constraints:
- `sm:col-span-2` — full-width row, matching "My Rig" and "My Antenna" layout
- Checkbox is 16px (w-4 h-4) — matches existing icon sizing convention
- Checkbox and span share a flex row with `gap-3` (12px) — compact, touch-friendly without excess padding
- `dark:bg-gray-800` on checkbox so the unchecked box is visible against the dark surface
- `dark:checked:bg-indigo-600` ensures the checked state uses accent color in dark mode
- Hidden input immediately follows checkbox (order matters — see RESEARCH.md Pitfall 1 note; implementation must verify Starlette Form() first-value behavior)

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | "Save Profile" (existing button — unchanged) |
| Section label | "Sound Notifications" |
| Checkbox span | "Play a tone when a new QSO arrives" |
| Hint text | "Requires at least one page interaction (browser autoplay policy)" |
| Empty state heading | not applicable — checkbox has two states only (checked/unchecked) |
| Empty state body | not applicable |
| Error state | not applicable — `bool` field cannot fail Pydantic validation; no user-visible error path |
| Destructive confirmation | none — this phase introduces no destructive actions |

Copywriting rationale:
- "Play a tone" is specific and action-oriented — avoids abstract "Enable sound notifications"
- "when a new QSO arrives" anchors the trigger to the concrete SSE event
- The hint text pre-empts the most common support question (why no sound on fresh tab) and sets correct expectations

Source: requirements SND-03/04/05 + RESEARCH.md template HTML example (adapted)

## Interaction Contract

### Checkbox State Machine

| State | Visual | Data |
|-------|--------|------|
| Unchecked (default for new users) | Empty checkbox, gray border | `notify_sound: false` in MongoDB |
| Checked | Indigo-filled checkbox with white checkmark | `notify_sound: true` in MongoDB |
| Page load (existing user with `notify_sound: true`) | Checkbox pre-filled via `{{ 'checked' if profile.notify_sound else '' }}` | Reads from `profile.notify_sound` passed by `profile_page()` |

### Form Submission Flow

1. Operator checks or unchecks the "Sound Notifications" checkbox
2. Operator clicks "Save Profile" (existing `btn-primary` submit button — no new button needed)
3. HTMX `hx-post="/log/profile"` sends form to `profile_update()` handler (existing wiring)
4. Handler receives `notify_sound` as `"true"` or `"false"` string via `Form()`
5. `raw["notify_sound"] = (notify_sound == "true")` converts to bool, stored unconditionally (not gated on `is not None`)
6. Success response swaps into `#profile-result` (existing HTMX target)
7. On page reload, checkbox reflects the saved state

No new HTMX targets, no new swap targets, no JavaScript introduced in this phase.
