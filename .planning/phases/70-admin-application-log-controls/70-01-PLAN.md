# Plan 70-01: Admin Application Log Controls

## Objective

Add Pause/Start live-feed control and confirmation-gated Clear Log Messages behavior to the admin Application Logs Recent Logs table. Preserve existing log capture, filters, pagination, retention settings, QSO data, and future logging. Cover the behavior with focused tests, docs, and UI source checks.

## Boundaries

In scope:

- Current-browser Pause/Start control for Recent Logs automatic SSE insertion and near-live polling fallback.
- Start/Resume immediately refreshes/reconciles recent rows without full page reload.
- Explicit filter, reset, and pagination actions continue to refresh the table while paused.
- Confirmation modal for clearing all stored application log messages.
- Clear execution deletes `ApplicationLog` records only, preserves `ApplicationLogSettings`, then attempts to create a forced audit record with deleted count.
- UI updates for LIVE/PAUSED/OFFLINE status and danger-outline Clear Log Messages button in the Recent Logs header.
- Admin docs and focused tests for pause/resume, clear safety, clear audit, and settings/data preservation.

Out of scope:

- Exporting or downloading logs.
- Clearing only filtered or date-range logs.
- Password-confirmed clear.
- Globally pausing log capture, SSE broadcast, polling, or all browser sessions.
- Deleting QSO records, users, API tokens, backups, or log settings.
- Broad redesign of the Application Logs page, filters, table, pagination, sidebar, or settings form.

## Threat Model

<threat_model>
### Assets

- Application log records and retention/settings configuration.
- Admin authentication/session cookies.
- QSO records, users, API tokens, backups, and unrelated MongoDB collections.
- Admin trust in the Recent Logs live/paused status.

### Threats And Mitigations

- **T-70-01: Clear action deletes unrelated data or settings.**
  Mitigation: isolate deletion to `ApplicationLog`; add tests/source checks proving `ApplicationLogSettings`, QSO collections, users, tokens, and backups are not targeted.

- **T-70-02: Clear audit row is removed by the clear operation.**
  Mitigation: delete first, then force-save `application_logs_cleared` audit event with deleted count.

- **T-70-03: Audit logging failure incorrectly fails the destructive clear after deletion.**
  Mitigation: preserve logger failure isolation; route reports clear success with a note when audit row save returns `None`.

- **T-70-04: Pause state gives false impression that logging stopped globally.**
  Mitigation: label/state and docs must describe current-browser live feed pause only; do not alter server storage, SSE broadcasting, or other browser sessions.

- **T-70-05: Paused state blocks explicit admin filter/pagination actions.**
  Mitigation: gate only automatic SSE insertion and polling interval, not `refreshLogsTable()` or HTMX controls.

- **T-70-06: Confirmation wording understates database-wide clear scope.**
  Mitigation: use locked copy: "Clear all application log messages from the database. QSO records, users, and log settings are not affected."

- **T-70-07: New UI controls degrade mobile/admin usability.**
  Mitigation: use compact wrapping header control group and existing admin design tokens; verify source/classes for responsive wrapping.
</threat_model>

## Success Criteria

- Admin can pause Recent Logs live updates for the current browser tab/session.
- Admin can start/resume live updates and immediately reconcile recent records without a page refresh.
- Paused state suppresses automatic SSE row insertion and near-live polling refreshes while preserving intentional filter/pagination updates.
- Admin can open a confirmation modal before clearing application log messages.
- Confirmed clear deletes all `ApplicationLog` records, preserves settings and unrelated data, and attempts a forced audit row with deleted count.
- Logs continue to be written and displayed after clear.
- Documentation and tests cover the new controls, clear scope, and safety guarantees.
