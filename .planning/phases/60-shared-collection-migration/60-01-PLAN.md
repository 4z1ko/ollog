# Plan 60-01: Idempotent Shared QSO Collection Migration

## Objective

Add an idempotent migration that copies legacy documents from the shared `qsos` collection into username-derived per-user QSO collections named `<username>_qsos`, preserving raw document data and reporting anything that cannot be safely routed.

## Boundaries

In scope:

- callable migration function
- CLI/report entry point
- startup integration
- unresolved/ambiguous operator reporting
- per-user index creation before writes
- focused migration tests

Out of scope:

- deleting or renaming the shared `qsos` collection
- changing QSO CRUD/import/export/API-token/UDP routing
- changing stats/admin/live-feed/backup/restore behavior
- resolving ambiguous legacy ownership by guessing

## Files Expected To Change

- `app/qso/collection_migration.py` - new migration module and CLI
- `app/main.py` - startup integration after source backfills
- `tests/test_qso_collection_migration.py` - focused migration tests
- `.planning/phases/60-shared-collection-migration/60-01-SUMMARY.md` - execution summary
- `.planning/phases/60-shared-collection-migration/60-VERIFICATION.md` - verification report

## Implementation Steps

1. Add the migration module.
   - Create `app/qso/collection_migration.py`.
   - Use the existing MongoDB client/database from `app.database.get_client()` and `settings.mongodb_db`.
   - Read from the legacy `qsos` collection.
   - Reuse Phase 59 helpers from `app.qso.collections` for target names and index setup.

2. Build user resolution.
   - Load all users from `User.find_all().to_list()`.
   - Build a callsign map keyed by normalized callsign.
   - A QSO is migratable only when `_operator` maps to exactly one user.
   - Missing, blank, unknown, and duplicate callsign mappings are reported as unresolved/ambiguous.
   - Do not rewrite the copied document's `_operator`.

3. Copy raw documents safely.
   - Copy each source document as a raw dict.
   - Preserve `_id` by using it in the upsert filter.
   - Use insert-only semantics such as `$setOnInsert` so an existing target `_id` is never overwritten.
   - Preserve `_operator`, `_deleted`, `_created_at`, `rowHash`, ADIF extras, profile-stamped fields, and custom fields.
   - Catch rowHash unique-index conflicts and report them as conflicts without deleting or overwriting data.

4. Ensure target indexes first.
   - Before writing to a user's collection, call `ensure_user_qso_indexes(...)`.
   - Cache initialized usernames during a run so indexes are created once per target collection.
   - Include initialized collection names in the migration report.

5. Return and write a useful report.
   - Include counts for scanned, migrated, already_present, unresolved, ambiguous, conflicts, failed, and collections_initialized.
   - Include enough row identifiers in unresolved/ambiguous/conflict entries for operator review.
   - Add report writing that mirrors the JSON style of `app/qso/row_hash_migration.py`.

6. Add a CLI.
   - Support `python -m app.qso.collection_migration`.
   - Provide `--report` with a sensible default path.
   - Provide `--dry-run` so operators can inspect unresolved/ambiguous ownership before writing target collections.
   - Initialize Beanie for `User`, `QSO`, and other required models without creating a second app lifecycle.

7. Wire startup integration.
   - In `app/main.py`, add a small wrapper such as `migrate_shared_qsos()`.
   - Run it after `backfill_row_hash()` so source rows have `rowHash` before copy.
   - Keep the source `qsos` collection in place because Phase 61 has not yet moved runtime CRUD.
   - Log a compact summary, including unresolved/ambiguous counts.

8. Add focused tests.
   - Normal migration copies documents to the expected username collections.
   - Rerun is idempotent and does not overwrite a target document with the same `_id`.
   - Unresolved operators are reported and not copied.
   - Duplicate callsign/user ambiguity is reported and not copied.
   - Soft-deleted documents are copied with `_deleted` preserved.
   - `rowHash`, `_created_at`, `_id`, ADIF extras, profile-stamped fields, and custom fields are preserved.
   - Target indexes are initialized before target writes.
   - Dry run reports intended work without writing target documents.

## Success Criteria

- Existing shared `qsos` documents are copied into the correct `<username>_qsos` collection for uniquely resolved users.
- Migration can rerun without creating duplicates or overwriting newer per-user collection data.
- Unresolved and ambiguous ownership is reported, not silently dropped.
- Raw QSO document data is preserved, including soft-delete state, rowHash, timestamps, ADIF extras, profile fields, and custom fields.
- Per-user collection indexes exist before migrated rows rely on them.
- Startup integration is idempotent and does not remove the shared collection.
- Focused tests pass without requiring a live MongoDB instance.

## Handoff To Later Phases

Phase 61 should assume that per-user collections may already contain migrated legacy data and should route all direct QSO workflows through the Phase 59 collection helpers. Phase 62 should extend backup/restore and cross-feature verification to include all dynamic QSO collections.
