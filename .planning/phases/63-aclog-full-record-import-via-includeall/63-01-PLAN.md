# Plan 63-01: ACLog INCLUDEALL Full-Record Import

## Objective

Extend the existing ACLog TCP bridge so each saved contact can be enriched from ACLog's `INCLUDEALL` full-record API response, preserving all non-empty fields ACLog exposes, including Other fields, while keeping the current `ENTEREVENT` import path as a reliable fallback.

## Boundaries

In scope:

- ACLog parser support for full-record `LIST INCLUDEALL`/related response payloads.
- Merge logic for compact `ENTEREVENT`, cached textbox state, and full-record responses.
- Bridge/client flow to request full-record data after saved-QSO events.
- Other field preservation and mapping to configured Custom QSO Fields.
- Deterministic parser/client tests that do not require live ACLog.
- Documentation updates for ACLog full-record behavior and fallback limits.

Out of scope:

- Replacing the ACLog TCP API bridge with another integration.
- Requiring live ACLog for automated CI/test success.
- Changing public QSO REST schemas or ADIF export semantics.
- Broad redesign of Profile Settings or ACLog bridge UI.

## Implementation Tasks

### 1. Add Full-Record Parser Helpers

**Files:**

- `app/aclog/parser.py`
- `tests/test_aclog_parser.py`

**Actions:**

- Add a parser helper for ACLog full-record responses returned by `LIST INCLUDEALL` or equivalent response messages.
- Preserve all non-empty fields whose tag names match the existing safe ADIF-like pattern.
- Normalize known aliases consistently with `aclog_enterevent_to_adif(...)`, including:
  - `RSTS` -> `RST_SENT`
  - `RSTR` -> `RST_RCVD`
  - numeric `BAND` -> ADIF-style `20M`
- Skip wrapper/control metadata such as counts, command echoes, and status-only tags.
- Keep `parse_cmd(...)` backward compatible with existing `ENTEREVENT`, `READBMFRESPONSE`, and `UPDATERESPONSE` tests.

**Acceptance criteria:**

- Full-record parser preserves arbitrary non-empty ADIF-like fields such as `POTA_REF`, `STATE`, `CNTY`, `GRIDSQUARE`, and `OTHER_1`.
- Empty fields are ignored.
- Existing parser tests still pass.

### 2. Add Deterministic Merge Logic

**Files:**

- `app/aclog/parser.py`
- `tests/test_aclog_parser.py`

**Actions:**

- Add `merge_aclog_records(base, full=None, state=None)` or similarly named helper.
- Merge precedence:
  1. Full-record non-empty fields win.
  2. Base `ENTEREVENT` fields fill anything missing.
  3. Cached state fills known live fields and Other slots when still missing.
- Preserve current band normalization and RST/FREQ behavior.
- Keep the function pure and easy to test without sockets or MongoDB.

**Acceptance criteria:**

- Tests prove full-record fields override compact event fields when non-empty.
- Tests prove event fields remain when full-record fields are missing.
- Tests prove state fallback still supplies `FREQ`, reports, and `OTHER_N`.

### 3. Request Full Record After ENTEREVENT

**Files:**

- `app/aclog/client.py`
- `tests/test_aclog_parser.py` or a new `tests/test_aclog_client.py`

**Actions:**

- After receiving `ENTEREVENT`, send a full-record request on the same writer:
  - Prefer `LIST INCLUDEALL VALUE 1` based on N3FJP API documentation.
- Add a small pending-enrichment state object that stores the base event record and correlation fields.
- When a full-record response arrives, parse it, verify it matches the pending event by stable fields (`CALL`, `QSO_DATE`, `TIME_ON`, and where available `BAND`/`MODE`), then ingest the merged record.
- If a full-record response does not arrive or does not match before the next event/reconnect, ingest the base event record with current cached-state behavior.
- Avoid blocking the socket read loop indefinitely.

**Acceptance criteria:**

- Bridge sends the `LIST INCLUDEALL VALUE 1` request after `ENTEREVENT`.
- Matching full-record response enriches the QSO before ingestion.
- Non-matching or missing full-record response falls back to current event-only ingestion behavior.
- Existing bridge reconnect behavior remains unchanged.

### 4. Preserve And Map Other Fields

- `tests/test_custom_qso_fields.py`

**Actions:**

- Keep `OTHER_1` through `OTHER_8` in the merged record when no custom field mapping exists.
- Continue mapping `OTHER_N` to configured `CustomQSOField.adif_name` after merge.
- Add parser support for `GETOTHERFIELDTITLES` response only if the observed/documented response shape can be represented safely in deterministic fixtures.
- Do not make title parsing a blocker for full-record import.

**Acceptance criteria:**

- Tests prove `OTHER_1` from full-record data maps to a configured field such as `POTA_REF`.
- Tests prove unmapped `OTHER_N` values remain preserved.
- Existing custom QSO field tests still pass.

## Threat Model

### Assets

- Per-user QSO records.
- Operator profile and custom field configuration.
- ACLog bridge host/port configuration.
- ACLog API data received over TCP.

### Threats And Mitigations

- **Malformed ACLog payload causes ingestion crash.**
  - Mitigation: parser helpers ignore unsafe tags, skip empty values, and fall back to event-only ingestion when full-record parsing/correlation fails.

- **Full-record response from a different QSO is merged into the saved event.**
  - Mitigation: require correlation on stable identity fields before enrichment.

- **Unexpected ACLog tag names inject unsafe MongoDB keys.**
  - Mitigation: preserve only safe ADIF-like uppercase alphanumeric/underscore keys; keep existing protected/custom-field normalization rules.

- **Bridge read loop stalls waiting for enrichment.**
  - Mitigation: use pending state and fallback rather than blocking indefinitely.

- **Cross-user QSO write regression.**
  - Mitigation: continue resolving the bridge owner by `config.user_id` and pass `get_user_qso_collection(user)` into `ingest_qso_record(...)`.
