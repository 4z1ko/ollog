## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#ffffff` light / `#1c1c1e` dark (`surface-light` / `surface-dark`) | Page background, modal box, table rows |
| Secondary (30%) | `#f9fafb` light / `rgba(31,41,55,0.6)` dark (`gray-50` / `gray-800/60`) | Table header, card header, hover row states |
| Accent (10%) | `#4f46e5` indigo-600 | Primary CTA buttons (Create Operator), focus rings |
| Destructive | `#e11d48` rose-600 | "Clear log" row button (`btn-danger btn-sm`), modal submit button (`btn-danger`), `alert-error` backgrounds |

Accent reserved for: the Create Operator submit button only. All other interactive elements in this phase use `btn-danger` (rose) or `btn-secondary` (gray).

Destructive color (`btn-danger` / rose-600) is used for:
1. The "Clear log" button in each operator's row actions
2. The "Delete N QSOs" submit button inside the confirmation modal

Dark mode: all classes use paired `dark:` variants already present in `input.css`. No new dark-mode classes needed.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Row action button label | `Clear log` |
| Row button `aria-label` | `Clear log for {username}` |
| Modal title | `Clear {callsign}'s Log` |
| Modal body (count > 0) | `This will permanently delete {N} QSO(s) for {callsign}. This cannot be undone.` |
| Modal body (count == 0) | `{callsign}'s log is empty (0 QSOs). Submitting your password will complete without deleting anything.` |
| Password field label | `Your admin password` |
| Password field placeholder | `Enter your password to confirm` |
| Modal submit button (count > 0) | `Delete {N} QSOs` |
| Modal submit button (count == 0) | `Confirm (0 QSOs)` |
| Modal cancel button | `Cancel` |
| Success message (count > 0) | `Done. {N} QSO(s) deleted from {callsign}'s log.` |
| Success message (count == 0) | `Done. {callsign}'s log was already empty — nothing was deleted.` |
| Inline password error | `Incorrect password. No QSOs were deleted.` |

**Destructive action confirmation approach:**
- Trigger: "Clear log" button in the row actions div fires `hx-get` to load the modal — no immediate action.
- Confirmation: Modal requires the admin to type their own password and click the submit button. Two-step: intent + credential.
- No browser `confirm()` dialog — consistent with existing admin restore and Phase 54 operator clear-log patterns.

## Component Inventory

All components are existing utility classes — no new CSS is authored for this phase.

### Row Action Button ("Clear log")

```
class="btn-danger btn-sm"
hx-get="/admin/ui/users/{{ user.username }}/clear-log/modal"
hx-target="#admin-clear-log-modal"
hx-swap="innerHTML"
aria-label="Clear log for {{ user.username }}"
```

Placement: appended at the end of the existing `<div class="flex flex-wrap items-center gap-2">` in `templates/admin/users_table.html`. Follows the existing Enable/Disable button and the Reset Password form.

Icon: trash icon (Heroicons outline `trash`), `w-4 h-4`, `aria-hidden="true"` — matches icon convention of adjacent row buttons.

### Modal Container Placeholder

```html
<div id="admin-clear-log-modal"></div>
```

Placement: OUTSIDE the `<table>` element in `templates/admin/users.html`, after the `<div class="card overflow-hidden">` block. This ensures `hx-swap="outerHTML"` on the modal fragment does not disturb table rows (established pattern per CONTEXT.md D-Cancel mechanism).

### Confirmation Modal Fragment (`templates/admin/clear_log_modal.html`)

Structure mirrors `templates/admin/restore/password_modal.html` exactly:

```
<div id="admin-clear-log-modal">
  <div class="modal-backdrop"></div>
  <div class="modal-box" role="dialog" aria-modal="true" aria-labelledby="admin-clear-log-title">
    <h3 id="admin-clear-log-title" class="modal-title">Clear {callsign}'s Log</h3>
    <p class="modal-body">...</p>
    [optional inline error: <div class="alert alert-error" role="alert">...</div>]
    <form hx-post="/admin/ui/users/{username}/clear-log"
          hx-target="#admin-clear-log-modal"
          hx-swap="outerHTML">
      <div class="form-group">
        <label for="admin-clear-log-password" class="form-label">Your admin password</label>
        <input id="admin-clear-log-password" type="password" name="password" required
               autocomplete="current-password" class="form-control"
               placeholder="Enter your password to confirm">
      </div>
      <div class="modal-actions">
        <button type="submit" class="btn btn-danger">Delete N QSOs</button>
        <button type="button" class="btn btn-secondary"
                hx-get="/admin/ui/users/{username}/clear-log/cancel"
                hx-target="#admin-clear-log-modal"
                hx-swap="outerHTML">Cancel</button>
      </div>
    </form>
  </div>
</div>
```

### Success Fragment (`templates/admin/clear_log_success.html`)

Mirrors `templates/log/clear_log_success.html` adapted for admin context:

```
<div id="admin-clear-log-modal">
  <div class="alert alert-success" role="alert">
    Done. {N} QSO(s) deleted from {callsign}'s log.
    [or: Done. {callsign}'s log was already empty — nothing was deleted.]
  </div>
</div>
```

Replaces modal via `outerHTML` swap — backdrop and modal-box disappear; inline success alert persists at the page-level placeholder position until next page load or another modal is opened.

### Password Error Inline State

On incorrect password, the route returns the modal fragment again (HTTP 200) with the error rendered inline between the modal body paragraph and the form:

```
<div class="alert alert-error" role="alert">Incorrect password. No QSOs were deleted.</div>
```

No toast, no page reload. Modal stays open with the error visible. Password field is empty (not pre-filled). Follows Phase 54 and admin restore error patterns.

### Cancel Route Response

`GET /admin/ui/users/{username}/clear-log/cancel` returns:

```
<div id="admin-clear-log-modal"></div>
```

Empty div with the same ID replaces the modal via `outerHTML` — backdrop and modal-box disappear without a page reload. Mirrors the admin restore cancel pattern.

## HTMX Interaction Flow

```
1. Page load
   users.html rendered with <div id="admin-clear-log-modal"></div> outside the table

2. Admin clicks "Clear log" for operator X
   → hx-get /admin/ui/users/X/clear-log/modal
   → hx-target="#admin-clear-log-modal" hx-swap="innerHTML"
   → Server returns modal HTML (fetches QSO count for X)
   → Modal appears: backdrop + modal-box

3a. Admin enters correct password and submits
    → hx-post /admin/ui/users/X/clear-log
    → hx-target="#admin-clear-log-modal" hx-swap="outerHTML"
    → Server deletes QSOs, returns success fragment (id="admin-clear-log-modal")
    → Modal replaced by inline alert-success

3b. Admin enters incorrect password and submits
    → hx-post /admin/ui/users/X/clear-log
    → hx-target="#admin-clear-log-modal" hx-swap="outerHTML"
    → Server returns modal fragment (id="admin-clear-log-modal") with alert-error inserted
    → Modal re-renders with error visible; modal stays open

4. Admin clicks Cancel
   → hx-get /admin/ui/users/X/clear-log/cancel
   → hx-target="#admin-clear-log-modal" hx-swap="outerHTML"
   → Server returns <div id="admin-clear-log-modal"></div>
   → Modal disappears; placeholder restored
```

All server responses use `status_code=200` — HTMX 2.x drops body on 4xx (per critical build rule).

## Accessibility

| Element | Requirement |
|---------|-------------|
| "Clear log" row button | `aria-label="Clear log for {username}"` (distinguishes per-row) |
| Modal outer div | `role="dialog" aria-modal="true" aria-labelledby="admin-clear-log-title"` |
| Modal title `<h3>` | `id="admin-clear-log-title"` (referenced by `aria-labelledby`) |
| Password input | `<label for="admin-clear-log-password">` with explicit `for`/`id` pairing |
| Alert fragments | `role="alert"` on `.alert-error` and `.alert-success` divs |
| Trash icon SVG | `aria-hidden="true"` — label on the button element carries the accessible name |
