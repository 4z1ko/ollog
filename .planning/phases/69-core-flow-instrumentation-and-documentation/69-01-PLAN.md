# Plan 69-01: Core Flow Instrumentation Reconciliation

## Objective

Reconcile Phase 69 against the logging instrumentation already shipped in Phase 67, then fill only the remaining OBS-01 through OBS-05 gaps. Add strict event-contract tests, internal logs for missing import/sync/auth/token outcomes, coverage-focused docs, and verification that existing QSO behavior remains unchanged.

## Boundaries

In scope:

- Preserve the existing internal logger API, MongoDB log storage model, admin viewer, filters, retention, and live stream behavior.
- Verify lifecycle, QSO, UDP, ACLog, auth/admin, and log-setting event contracts.
- Add missing internal logs for manual ACLog sync outcomes.
- Add missing internal logs for ADIF import outcomes.
- Add missing safe auth/token action logs where current operator or token flows are uncovered.
- Fix any async logger dispatch defect found in UDP/ACLog callback paths.
- Update admin logging documentation with a compact coverage matrix and field-safety notes.
- Add focused tests that prove event names, source/module, transport, and safe metadata for representative flows.

Out of scope:

- Renaming or normalizing existing event names without a concrete defect.
- Reworking the admin Logs page UI from Phase 68.
- Changing QSO ownership, duplicate detection, row hashing, collection routing, or ACLog identity matching behavior.
- Logging raw ADIF payloads, UDP datagrams, passwords, tokens, authorization headers, cookies, API-key hashes, or full connection strings.
- Writing a scenario-based troubleshooting runbook.

## Threat Model

<threat_model>
### Assets

- Application log records stored in MongoDB and rendered in the admin UI.
- Operator credentials, JWTs, API tokens, API-token hashes, cookies, and authorization headers.
- QSO records, ADIF import payloads, UDP datagrams, and ACLog TCP messages.
- QSO ownership/routing, duplicate detection, per-user collections, and ACLog identity matching.

### Threats And Mitigations

- **T-69-01: Sensitive data leakage through new instrumentation.**
  Mitigation: log only safe identifiers and counts; never log passwords, bearer tokens, plaintext API tokens, API-token hashes, cookies, authorization headers, raw ADIF files, raw UDP datagrams, or raw ACLog TCP payloads. Add tests that assert forbidden metadata keys are absent.

- **T-69-02: Logging failure breaks QSO ingestion or auth/admin flows.**
  Mitigation: use existing failure-isolated `app_logger`; do not make business success depend on log insert results; preserve existing exception handling and return shapes.

- **T-69-03: Instrumentation changes QSO behavior.**
  Mitigation: do not change ownership/routing, duplicate detection, row hashing, collection selection, or ACLog identity matching. Run existing focused QSO/UDP/ACLog tests after implementation.

- **T-69-04: Incorrect local-station attribution in logs.**
  Mitigation: keep ADIF semantics explicit: `CALL` is contacted station, `MYCALL` or ACLog setup Call is local station, `OPERATOR` is operator value. Tests must prove contacted-station `CALL` is not treated as local station identifier.

- **T-69-05: Noisy logs reduce operational usefulness.**
  Mitigation: use `Info` for routine lifecycle/import summaries, `Warn` for rejected/skipped/retry outcomes, and `Error` only for operation failures requiring attention. Avoid per-record raw payload dumps.

- **T-69-06: Async logger coroutines are created but not scheduled or awaited.**
  Mitigation: review all new and touched log calls; use `await` inside async functions and `asyncio.create_task()` only from synchronous protocol callbacks. Add regression tests around callback-emitted UDP events.
</threat_model>
