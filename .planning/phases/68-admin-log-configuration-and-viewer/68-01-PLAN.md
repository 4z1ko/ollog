# Plan 68-01: Admin Log Configuration and Viewer Reconciliation

## Objective

Reconcile Phase 68 against the admin log configuration/viewer work already shipped in Phase 67, then close the remaining acceptance gaps with compact Previous/Next pagination controls and readable collapsed JSON metadata/error details. Preserve existing admin authentication, settings, filters, live SSE insertion, documentation, and styling patterns.

## Boundaries

In scope:

- Verify and preserve existing admin log settings UI and JSON settings API.
- Verify and preserve level/source/text/date filtering.
- Add visible simple Previous/Next pagination controls to the admin Logs table.
- Preserve current immediate live insertion behavior for matching SSE log events.
- Format metadata and error payloads as readable JSON in collapsed details.
- Add/update focused tests for pagination query behavior, table context, and JSON formatting helpers.
- Keep admin help text for the six log levels and default `Info` behavior.

Out of scope:

- Numbered pagination.
- A broader admin Logs redesign.
- New UI dependencies, shadcn blocks, charting, or icon libraries.
- Changing log storage semantics from Phase 67.
- Changing live update behavior to a refresh prompt.
- New instrumentation for Phase 69 flows.

## Threat Model

<threat_model>
### Assets

- Admin-only operational log records and structured metadata.
- Log settings controlling minimum level and retention.
- Admin session cookies used to access `/admin/ui/logs` and `/admin/ui/logs/events`.
- Sanitized metadata/error details rendered into the Logs table.

### Threats And Mitigations

- **T-68-01: Unauthorized log viewing through UI pagination or live stream.**
  Mitigation: preserve `require_admin_cookie` on `/admin/ui/logs` and `/admin/ui/logs/events`; do not add unauthenticated routes or client-only access checks.

- **T-68-02: Sensitive metadata disclosure through formatted JSON.**
  Mitigation: reuse Phase 67 sanitized `metadata` and `error` fields; formatting must not rehydrate hidden data, bypass sanitization, or fetch raw source records.

- **T-68-03: XSS through metadata/error details.**
  Mitigation: render JSON through Jinja text escaping inside `<pre>`; do not mark metadata/error JSON as safe HTML.

- **T-68-04: Filter or pagination links drop admin's current constraints and show broader logs unexpectedly.**
  Mitigation: pagination links/HTMX requests preserve `level`, `source`, `search`, `date_from`, and `date_to`.

- **T-68-05: Live SSE row insertion bypasses filter semantics.**
  Mitigation: preserve and test existing client-side `matchesFilters(log)` behavior for level/source/search/date filtering before insertion.
</threat_model>
