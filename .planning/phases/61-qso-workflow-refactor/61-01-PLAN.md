# Plan 61-01: Dynamic QSO Workflow Collection Routing

## Objective

Move direct QSO runtime workflows from Beanie's fixed shared `qsos` collection to the authenticated or resolved user's `<username>_qsos` collection while keeping public API, browser, ADIF, API-token, and UDP behavior unchanged.

## Boundaries

In scope:

- REST QSO CRUD routing
- browser QSO entry/log view/edit/delete/import/export/duplicate review/clear-log routing
- ADIF REST import/export routing
- API-token QSO routing
- UDP QSO ingestion routing
- duplicate detection and rowHash uniqueness scoped to user collection
- custom QSO default lookup from user collection
- focused compatibility/isolation tests

Out of scope:

- stats aggregation
- admin clear-log
- live feed/SSE watcher changes
- backup/restore changes
- dropping or archiving the shared `qsos` collection

## Files Expected To Change

- `app/qso/service.py`
- `app/qso/router.py`
- `app/qso/ui_router.py`
- `app/qso/custom_fields.py`
- `app/adif/router.py`
- `app/udp/server.py`
- tests covering service helpers and representative workflows
- `.planning/phases/61-qso-workflow-refactor/61-01-SUMMARY.md`
- `.planning/phases/61-qso-workflow-refactor/61-VERIFICATION.md`

## Implementation Steps

1. Add collection-aware QSO service primitives.
   - Add helpers to hydrate raw MongoDB docs into `QSO` objects.
   - Add helpers to serialize validated `QSO` objects into raw MongoDB documents with aliases/extras preserved.
   - Ensure insert computes `rowHash` exactly as existing `QSO.insert()` did.
   - Add collection-aware get/list/update/soft-delete/delete-many helpers.
   - Keep `QSO` as the response/template validation shape.

2. Refactor duplicate and insert behavior.
   - Update `find_duplicate()` to query a supplied user collection.
   - Update `insert_qso_dict()` to insert into a supplied user collection.
   - Keep duplicate conflict reporting compatible with existing REST/UI responses.
   - Scope rowHash lookup to the same collection on duplicate-key handling.

3. Refactor pagination and clear-log service behavior.
   - Update `get_qso_page()` to query the user's collection with the existing filters/sort allowlist.
   - Preserve total count semantics and sort fallback logging.
   - Update operator clear-log to delete from the logged-in user's collection while preserving count return behavior.

4. Refactor REST QSO routes.
   - Use `get_current_user_jwt_or_apikey` for list/get/create/patch/delete where storage access is needed.
   - Derive the collection from `user.username`.
   - Continue stamping `_operator` from `user.callsign`.
   - Preserve route paths, status codes, response schemas, duplicate 409 payloads, and protected field stripping.

5. Refactor browser QSO routes.
   - Use `get_current_user_cookie` for QSO storage routes that currently take callsign only.
   - Move QSO entry duplicate/insert, log view, edit row, cancel row, inline update, delete, import, duplicate review, export, and clear-log to the user's collection.
   - Preserve all templates, context keys, HTMX partials, and HTTP 200 error-fragment behavior.
   - Update `custom_field_defaults()` to query the user's collection.

6. Refactor ADIF REST routes.
   - Use authenticated `User` instead of callsign-only dependency.
   - Route import duplicate checks/inserts and export reads through the user's collection.
   - Preserve import/export schemas and streamed ADIF format.

7. Refactor UDP ingestion.
   - Update `ingest_qso_record()` or its callers so it receives a `User` or target collection.
   - Token path writes to token owner's collection.
   - OPERATOR override writes to resolved operator user's collection.
   - UDP fallback writes only when startup resolved `UDP_OPERATOR` to a user; otherwise reject/log clearly.
   - Preserve APP_OLLOG_TOKEN consumption so tokens are not stored in QSO documents.

8. Update tests.
   - Add `tests/test_qso_service_collections.py` for raw collection helper behavior, duplicate scope, rowHash conflicts, pagination, and clear-log.
   - Update existing REST/API-key/ADIF/UDP tests to assert per-user collection routing where practical.
   - Keep Mongo-dependent tests skipped when MongoDB is unavailable, but make core service behavior testable without live MongoDB.
   - Cover same effective QSO in two user collections as allowed.
   - Cover another user's QSO ID returning not found from REST/UI paths.

9. Run source review.
   - Search for remaining Phase-61-scope `QSO.find`, `QSO.get`, `QSO.find_one`, `.insert()`, and `QSO.get_pymongo_collection()` usage.
   - Leave Phase-62-owned occurrences in stats/admin/live-feed/backup noted but not refactored.
   - Confirm no route accepts collection name, username, or callsign from request data for storage routing.

## Success Criteria

- REST QSO CRUD targets the authenticated user's collection with unchanged API contracts.
- Browser QSO workflows target the logged-in user's collection with unchanged templates/HTMX behavior.
- ADIF import/export and duplicate review use the user's collection.
- API-token QSO requests target the token owner's collection.
- UDP writes target the resolved user's collection.
- Duplicate detection and rowHash uniqueness are scoped per user collection.
- `_operator` remains present and stores callsign in QSO documents.
- Representative tests prove isolation and compatibility.
