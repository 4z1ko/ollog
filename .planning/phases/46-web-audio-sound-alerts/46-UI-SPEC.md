## JS Behavior Contract

This section replaces the typical visual layout contract for this phase. It is the
authoritative specification for the audio subsystem and NOTIFY_SOUND wiring.

### Constants and State Variables

Declared inside the existing IIFE `<script>` block in `templates/log/log.html`,
alongside `var indicator` and `var eventsFlowing`:

| Variable | Type | Initial Value | Scope | Purpose |
|----------|------|---------------|-------|---------|
| `NOTIFY_SOUND` | `const string` | `"true"` or `"false"` (SSR-injected) | Outer `<script>` block, before IIFE | Preference flag injected by Jinja2; compared as string |
| `AudioCtxClass` | `var` | `window.AudioContext \|\| window.webkitAudioContext` | IIFE scope | Browser-compat constructor reference |
| `audioCtx` | `var` | `null` | IIFE scope | AudioContext instance; created lazily on first gesture |
| `userInteracted` | `var` | `false` | IIFE scope | True after first `click` or `keydown` on `document` |

**Jinja2 injection (D-04):** The constant must be injected as:

```html
const NOTIFY_SOUND = "{{ 'true' if notify_sound else 'false' }}";
```

Never use `{{ notify_sound }}` directly — Python `True`/`False` booleans do not match
the JavaScript string `"true"`. The explicit conditional is mandatory. Source: CONTEXT.md D-04, RESEARCH.md Pitfall 3.

### Autoplay Policy State Machine

| State | `userInteracted` | `audioCtx` | Trigger | Outcome |
|-------|-----------------|-----------|---------|---------|
| Initial | `false` | `null` | Page load | No AudioContext created |
| First gesture | `true` | AudioContext instance | First `click` or `keydown` on `document` | AudioContext created, ready |
| Tone eligible | `true` | non-null | `new_qso` SSE event + `NOTIFY_SOUND === "true"` | `playTone(audioCtx)` called fire-and-forget |
| Pre-interaction | `false` | `null` | `new_qso` SSE event | Fail silently — no tone, no visual indicator |
| Sound disabled | any | any | `new_qso` SSE event + `NOTIFY_SOUND !== "true"` | Fail silently |
| No AudioContext | `true` | `null` | Very old browser (`AudioCtxClass` is `undefined`) | Fail silently |

**Source:** CONTEXT.md D-02, STATE.md §v2.4 Architecture Decisions.

### Interaction Listeners

| Event | Target | Handler | Effect |
|-------|--------|---------|--------|
| `click` | `document` | `onFirstInteraction()` | Sets `userInteracted = true`; creates `audioCtx` if `AudioCtxClass` available |
| `keydown` | `document` | `onFirstInteraction()` | Same as click |

`onFirstInteraction()` must guard itself: `if (userInteracted) return;` — idempotent.

**Source:** CONTEXT.md D-02.

### Tone Specification (D-01)

| Parameter | Value | Rationale |
|-----------|-------|-----------|
| Waveform | `sine` | Smooth, unobtrusive during long FT8 sessions |
| Frequency | 440 Hz | Concert A — immediately recognizable, not alarming |
| Total duration | 120 ms | Brief enough to not interrupt SSB/CW operating |
| Attack | 0 → 0.3 gain over 10 ms (`now` to `now + 0.01`) | Eliminates click artifact at start |
| Sustain | 0.3 gain held from 10 ms to 80 ms (`now + 0.01` to `now + 0.08`) | Audible portion |
| Decay | 0.3 → 0 gain over 30 ms (`now + 0.08` to `now + 0.11`) | Smooth fade-out |
| oscillator.stop() | `now + 0.12` | 10 ms buffer after decay end — avoids click from stopping mid-ramp |
| Peak gain | 0.3 | Moderate volume; not startling; no output clipping |

**AudioParam scheduling:** Use `linearRampToValueAtTime` exclusively. Never set `gain.value` directly during playback — direct assignment causes clicks at transition boundaries. Source: RESEARCH.md Anti-Patterns.

**Node graph:** `OscillatorNode` → `GainNode` → `ctx.destination`. Nodes are created per-tone, not reused. Source: RESEARCH.md Pattern 2.

### Suspended AudioContext Handling

Before scheduling any AudioParam ramp or starting the oscillator, check and resume:

```
if (ctx.state === 'suspended') { await ctx.resume(); }
```

`playTone()` must be declared `async` to use `await ctx.resume()`. The caller (`htmx:sseMessage` listener) must call `playTone(audioCtx)` without `await` (fire-and-forget) — the outer listener remains a synchronous `function`. Source: RESEARCH.md Pitfall 2, STATE.md §v2.4 Architecture Decisions.

### SSE Event Guard Ordering in `htmx:sseMessage`

The tone playback code must be placed **before** the `auto-refresh-ok` guard. Existing
code has an early `return` at line 137 when `#auto-refresh-ok` is absent (operator is
on page 2+). If the tone block is placed after that `return`, the tone will never play
while browsing paginated results — violating SND-01. The correct ordering:

```
1. [existing guard] if (!e.target || e.target.id !== 'log-table') return;
2. [existing guard] if (!e.detail || e.detail.type !== 'new_qso') return;
3. [existing] LIVE indicator update
4. [NEW Phase 46] playTone() call — here, before the auto-refresh guards
5. [existing guard] if (!document.getElementById('auto-refresh-ok')) return;
6. [existing guard] if (document.querySelector('#log-table input')) return;
7. [existing] htmx.ajax() reload
```

**Source:** RESEARCH.md Open Question / Assumption A2 — resolved: tone is independent of auto-refresh state.

### webkitAudioContext Fallback

```javascript
var AudioCtxClass = window.AudioContext || window.webkitAudioContext;
```

Required for Safari < 14.1. If neither is defined (very old browser), `AudioCtxClass`
is `undefined` — `onFirstInteraction()` must check before constructing:
`if (AudioCtxClass) { audioCtx = new AudioCtxClass(); }`. Source: CONTEXT.md, STATE.md.

### Backend Wiring (D-04)

`app/qso/ui_router.py` — `log_view()` signature change:

| Before | After |
|--------|-------|
| `callsign: str = Depends(get_current_operator_callsign_cookie)` | `user: User = Depends(get_current_user_cookie)` |

First line of handler body: `callsign = user.callsign`

Template context addition: `"notify_sound": user.notify_sound`

HTMX partial path (D-05): When `request.headers.get("HX-Request")` is truthy, `log_view()` returns `log_table.html`. The `notify_sound` key will be present in `ctx` but is harmlessly ignored by `log_table.html`. No conditional injection needed.

**Source:** CONTEXT.md D-04, D-05; RESEARCH.md Pattern 4.
