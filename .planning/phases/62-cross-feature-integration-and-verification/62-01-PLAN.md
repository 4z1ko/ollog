# Plan 62-01: Cross-Feature Dynamic Collection Integration

## Objective

Finish v3.1 by moving remaining cross-feature QSO paths onto per-user `<username>_qsos` collections, replacing the single shared live-feed watcher with app-level broadcasts from supported write paths, proving backup/restore includes dynamic collections, and adding layered isolation/regression coverage.

## Boundaries

In scope:

- `/log/stats` aggregation and route wiring
- admin clear-log modal/count/delete routing
- live feed/SSE broadcast behavior for app-created QSO inserts
- backup/restore tests for dynamic QSO collections
- representative cross-feature isolation/regression tests
- requirement traceability updates for v3.1

Out of scope:

- MongoDB change streams across every user collection
- live feed broadcasts for manual database inserts, restore operations, or migration/backfill events
- new admin migration status UI
- username rename tooling
- removing the legacy shared `qsos` collection

## Threat Model

### Assets

- Per-user QSO logs in `<username>_qsos` collections.
- Admin credentials and clear-log authority.
- Backup files containing all operator data.
- Live feed event stream.

### Threats and Mitigations

- **Cross-user data exposure:** Stats/admin/feed must not read the wrong collection. Mitigate by deriving target collections only from authenticated or DB-loaded `User` objects.
- **Unauthorized destructive delete:** Admin clear-log must verify the admin's own password. Preserve existing `verify_password(password, current_user.hashed_password)` behavior.
- **Request-controlled collection selection:** Never accept collection names or usernames from request bodies for storage routing. URL username in admin routes only identifies a DB-loaded `User`.
- **Backup omission:** Full-database backup avoids missing dynamic QSO collections. Tests must protect this.
- **Event leakage:** Current feed is a shared station feed by product design. Preserve existing behavior, but do not expand feed payload beyond current row data.
