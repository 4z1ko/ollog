**docs/admin-guide/admin-container.md** — NEW page (DOC-04):

Cover:
- What the admin container is: a separate FastAPI app (`app.admin_main:app`) sharing the same Docker image but exposing admin-only routes. Runs on port 8001.
- Starting the admin container: `docker compose --profile admin up -d admin`. It uses `profiles: [admin]` in docker-compose.yml — it does NOT start with plain `docker compose up`.
- Stopping: `docker compose --profile admin stop admin` or `docker compose --profile admin down` (does not affect the operator app on port 8000).
- Routes exposed: `/admin/*` (user management), `/auth` (admin login), `/health`.
- Auth: the admin UI browser session uses the `admin_token` cookie (NOT `access_token` — the names differ to prevent cookie collision between port 8000 and port 8001 per RFC 6265). Admin REST API calls use `Authorization: Bearer <jwt>`.
- First login: navigate to `http://hostname:8001/`, log in with ADMIN_USERNAME/ADMIN_PASSWORD from .env.
- Security note: do NOT expose port 8001 publicly. Use a firewall or reverse proxy to restrict access to trusted networks.

**docs/admin-guide/backup.md** — NEW page (DOC-05):

Cover:
- What the backup CLI does: exports the entire MongoDB database as EJSON-encoded gzip file to BACKUP_DIR (default /app/backups). Uses PyMongo `bson.json_util.dumps()` — no mongodump required.
- Running a manual one-shot backup: `docker compose exec app python -m app.backup` (or `python -m app.backup` if running outside Docker). Prints the path of the created .gz file.
- The bind mount: `./backups:/app/backups` in docker-compose.yml means backup files are written to the host `./backups/` directory and survive container restarts.
- Automated scheduled backups: set `BACKUP_SCHEDULE` env var to a cron expression. Example: `BACKUP_SCHEDULE=0 2 * * *` (nightly at 02:00 UTC). The scheduler does not start if `BACKUP_SCHEDULE` is absent.
- S3 upload: if `BACKUP_S3_BUCKET` is set, the backup file is uploaded to S3 after creation. Uses standard boto3/aioboto3 credential chain (env vars, ~/.aws/credentials, IAM role). Required env vars: `BACKUP_S3_BUCKET`. Optional: `BACKUP_S3_PREFIX` (default "backups/").
- Environment variables table for backup:
  | Variable | Required | Default | Description |
  |----------|----------|---------|-------------|
  | BACKUP_DIR | No | /app/backups | Local directory for backup files |
  | BACKUP_SCHEDULE | No | (none) | Cron expression; scheduler not started if absent |
  | BACKUP_S3_BUCKET | No | (none) | S3 bucket for upload; upload skipped if absent |
  | BACKUP_S3_PREFIX | No | backups/ | S3 key prefix |
  | AWS_ACCESS_KEY_ID | No* | — | AWS credentials (boto3 credential chain) |
  | AWS_SECRET_ACCESS_KEY | No* | — | AWS credentials (boto3 credential chain) |
  | AWS_DEFAULT_REGION | No* | — | AWS region (required for S3) |
  (* Required when using S3 upload without IAM role)
- Restoring: copy the .gz file out of the container / S3; restore with `mongorestore --gzip --archive=<file>` against the running MongoDB container.

**docs/operator-guide/api-tokens.md** — NEW page (DOC-06):

Cover:
- What API tokens are: long-lived tokens for programmatic / automation access. Unlike JWT sessions (15-480 min, browser login), API tokens are revocable and named, suited for scripts and integrations.
- Creating a token: POST /api/tokens/ (requires JWT auth). Fields: `name` (required), `expires_at` (optional, YYYY-MM-DD). The full token value is returned ONCE on creation — copy it immediately.
  ```bash
  curl -X POST http://localhost:8000/api/tokens/ \
    -H "Authorization: Bearer $JWT" \
    -F "name=my-script" \
    -F "expires_at=2025-12-31"
  ```
- Listing tokens: GET /api/tokens/ (returns name, id, created_at, expires_at — NOT the token value)
  ```bash
  curl http://localhost:8000/api/tokens/ -H "Authorization: Bearer $JWT"
  ```
- Revoking a token: DELETE /api/tokens/{token_id}
  ```bash
  curl -X DELETE http://localhost:8000/api/tokens/{token_id} -H "Authorization: Bearer $JWT"
  ```
- Using a token in REST API calls: `X-API-Key: ollog_<token>` header. All endpoints that accept JWT auth also accept X-API-Key.
  ```bash
  curl http://localhost:8000/api/qsos/ -H "X-API-Key: ollog_abc123..."
  ```
- Using a token in UDP ADIF datagrams: include the `APP_OLLOG_TOKEN` ADIF field in the datagram. This is an ADIF field name (APP_ prefix per ADIF spec), NOT an environment variable. When present, ollog validates the field value against the API token store.
  Example ADIF datagram with token: `<APP_OLLOG_TOKEN:20>ollog_abc123...<CALL:4>W1AW<...><EOR>`
- The profile page at /profile also has a "Manage API Tokens" panel with the same create/list/revoke functionality.
