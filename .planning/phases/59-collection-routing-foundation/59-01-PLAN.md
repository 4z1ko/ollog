# Plan 59-01: Dynamic Collection Helper and Index Foundation

## Objective

Create one safe, tested foundation for username-routed QSO storage. After this plan, later phases can obtain a raw MongoDB collection for the logged-in user's `<username>_qsos` collection without depending on Beanie's fixed `QSO.Settings.name = "qsos"` CRUD surface.

## Boundaries

In scope:

- add shared collection-name derivation
- add safe dynamic collection access through the existing MongoDB client
- add idempotent per-user QSO index setup
- add focused unit tests

Out of scope:

- migrating existing shared `qsos` documents
- changing REST/UI/ADIF/API-token/UDP workflows
- changing stats/admin/live-feed/backup behavior
- removing `_operator` from QSO documents

## Files Expected To Change

- `app/qso/collections.py` - new dynamic collection helper module
- `tests/test_qso_collections.py` - new focused tests
- `tests/test_qso_schema.py` - update schema/index expectations only if needed

## Implementation Steps

1. Add `app/qso/collections.py`.
   - Implement `qso_collection_name(username: str) -> str`.
   - Preserve safe usernames exactly before appending `_qsos`.
   - Reject empty, non-string, whitespace-padded, separator-containing, null-byte, `$`, dot, and reserved `system` namespace inputs.
   - Use a conservative allowlist such as ASCII letters, digits, underscore, and hyphen.

2. Add dynamic collection access.
   - Implement `get_qso_collection_for_username(username: str)`.
   - Use `app.database.get_client()` and `settings.mongodb_db`.
   - Raise a clear `RuntimeError` if the database client has not been initialized.
   - Implement a convenience helper that accepts a `User` object and routes through `user.username`.

3. Add index definitions and setup.
   - Define dynamic QSO `IndexModel` objects in the helper module.
   - Mirror the current `QSO.Settings.indexes` intent:
     - `operator_qso_compound`
     - `operator_idx`
     - `operator_active_idx`
     - `operator_created_at_idx`
     - `row_hash_unique_idx`
   - Preserve unique sparse `rowHash` semantics.
   - Implement `ensure_user_qso_indexes(collection)` and `ensure_user_qso_collection_indexes(user_or_username)`.
   - Keep setup idempotent by delegating to MongoDB `create_indexes(...)`.

4. Add focused tests.
   - Prove `john_doe` maps to `john_doe_qsos`.
   - Cover representative safe usernames.
   - Cover unsafe usernames and reserved names.
   - Monkeypatch `get_client()` or use small fakes to prove database/collection indexing behavior.
   - Assert the index model names and unique/sparse rowHash options.

5. Run source review.
   - Confirm no production call sites were accidentally moved into mixed routing during this foundation phase.
   - Confirm the helper module can be imported by Phase 60 migration and Phase 61 workflow refactors.

## Success Criteria

- A shared helper derives collection names exactly as `<username>_qsos`.
- Unsafe usernames fail loudly before becoming MongoDB collection names.
- Runtime code can obtain the correct raw MongoDB collection for a `User.username`.
- Per-user QSO collections can receive all indexes needed by current QSO behavior.
- `QSO` remains available as the validation/serialization model while storage routing moves toward raw dynamic collections.
- Focused tests pass without requiring a live MongoDB instance.

## Handoff To Later Phases

Phase 60 should use these helpers to create indexes before migrating each user's data from the legacy shared `qsos` collection. Phase 61 should route authenticated QSO CRUD and import/export paths through these helpers. Phase 62 should extend cross-feature integrations such as stats, admin clear-log, live feed, and backup/restore.
