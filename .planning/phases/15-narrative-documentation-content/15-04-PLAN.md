**docs/adif-field-reference.md** (DOCS-06):

# ADIF Field Reference

Brief intro: ollog uses ADIF (Amateur Data Interchange Format) for QSO data. This page describes how key fields are formatted and handled.

**## Core Fields**

Table format:

| Field | Format | Example | Notes |
|-------|--------|---------|-------|
| QSO_DATE | YYYYMMDD | 20240415 | UTC date of contact |
| TIME_ON | HHMM or HHMMSS | 1430 or 143045 | UTC time of contact |
| CALL | Callsign string | DL1ABC | Station worked |
| BAND | Amateur band designator | 20m, 40m, 2m | Standard ADIF band values |
| MODE | Uppercased mode | SSB, CW, FT8, FM | Standard ADIF mode values |
| RST_SENT | Signal report string | 59, 599 | 599 for CW, 59 for phone |
| RST_RCVD | Signal report string | 59, 599 | Received signal report |

**## Auto-Stamped Fields**

| Field | Source | When Stamped | Notes |
|-------|--------|--------------|-------|
| OPERATOR | JWT callsign (account) | Every QSO created via API/UI | Always stamped. Cannot be overridden by the user. |
| STATION_CALLSIGN | Profile station_callsign | When profile has station_callsign set | Optional. Represents the station being operated (e.g., club call). |

Explain: When importing ADIF files, OPERATOR and STATION_CALLSIGN are NOT auto-stamped. The values from the file are preserved as-is. This ensures historical records maintain their original attribution.

**## Extra Fields**

Explain: ollog accepts any standard ADIF field beyond the core set. Extra fields are stored as-is in the QSO record and included in ADIF exports. The QSO creation endpoint uses `extra="allow"` on its Pydantic model -- any additional fields in the JSON body are accepted and persisted.

Examples of extra fields: FREQ, TX_PWR, COMMENT, QTH, GRIDSQUARE, CONTEST_ID, SRX, STX

**## Duplicate Detection**

Explain the detection algorithm:
- Window: +/- 2 minutes on QSO_DATE + TIME_ON
- Match fields: CALL + BAND + MODE + operator (the logged-in user)
- Response: 409 Conflict with error details
- Override: append `?force=true` to the POST URL

# Troubleshooting

**## SSE Station Feed Not Updating**

Symptom: You log a QSO but the live feed does not show it.

Cause: MongoDB is not running in replica set mode. ollog uses MongoDB change streams for the live feed, and change streams require a replica set.

Fix:
1. Check that MongoDB was started with `--replSet rs0` (the docker-compose.yml does this automatically)
2. Verify replica set is initialized: `docker compose exec mongodb mongosh --eval "rs.status()"` -- should show a member with stateStr "PRIMARY"
3. Check that MONGODB_URI includes `?replicaSet=rs0` (the docker-compose.yml sets this)
4. If using an external MongoDB, ensure it is configured as a replica set

**## Login Fails After Container Restart**

Symptom: Users cannot log in after restarting the containers, even with correct credentials.

Cause: The SECRET_KEY was not set explicitly in `.env`. The docker-compose.yml sets a default `dev-secret-change-in-production`, but if this value changes (e.g., by removing the environment line or changing compose files), all previously issued JWTs become invalid and the password hashing may differ.

Fix:
1. Set `SECRET_KEY` to a fixed value in your `.env` file
2. If passwords were hashed with a different key, you may need to reset passwords via the admin API

Note: SECRET_KEY is used for JWT signing. It does not affect password hashing (passwords use bcrypt). The actual symptom is that existing tokens stop working. Users can still log in with username/password to get a new token. If login itself fails with correct credentials, the issue is likely a different problem (account disabled, wrong password).

IMPORTANT: Read the actual auth code to verify whether SECRET_KEY affects password hashing before writing this section. If it does not (bcrypt is independent of SECRET_KEY), adjust the explanation accordingly. The most likely scenario for "login fails after restart" is that the old JWT cookie in the browser is invalid -- clearing cookies or re-logging fixes it.

**## ADIF Import Returns All Duplicates**

Symptom: Importing an ADIF file reports every record as a duplicate.

Cause: The file was already imported previously. ollog's duplicate detection matches on CALL + BAND + MODE + operator within a +/- 2 minute window, so re-importing the same file will flag every record.

Fix:
- If you intentionally want to re-import, delete the existing records first (via the UI or API) and import again
- If you want to force-import despite duplicates, note that the import endpoint does NOT support `?force=true` -- force is only available on single QSO creation. For bulk re-import, delete first then import.

Read the actual import endpoint code to verify whether force=true is supported on the import endpoint. If it IS supported, update the fix accordingly.
