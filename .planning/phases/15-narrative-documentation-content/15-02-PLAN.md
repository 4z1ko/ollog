**# Getting Started**

Brief: This walkthrough takes you through the complete ollog workflow. You need a running ollog instance and an operator account (see Deployment guide).

**## Step 1: Log In**

Two options:
- **Browser:** Navigate to `http://localhost:8000`, enter username and password
- **API:** Show curl command:
  ```bash
  TOKEN=$(curl -s -X POST http://localhost:8000/auth/token \
    -H "Content-Type: application/x-www-form-urlencoded" \
    -d "username=youruser&password=yourpass" | jq -r .access_token)
  ```
  Explain: The token is valid for JWT_EXPIRE_MINUTES (default 60 minutes). For browser sessions, a cookie is set automatically.

**## Step 2: Set Up Your Profile**

Explain OPERATOR vs STATION_CALLSIGN:
- OPERATOR: Your personal callsign. This is auto-stamped on every QSO you create. It comes from your account's callsign field (set by admin at account creation).
- STATION_CALLSIGN: The station you are operating FROM (e.g., a club call or special event call). This is optional and set via your profile.

Show how to set profile via API:
```bash
curl -X PATCH http://localhost:8000/api/profile/ \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"station_callsign": "W1AW", "grid_square": "FN31pr", "name": "Your Name"}'
```

Explain: If station_callsign is set, it auto-stamps new QSOs with STATION_CALLSIGN in addition to OPERATOR. You can change or clear it anytime.

**## Step 3: Log a QSO via the Web UI**

Walk through:
1. Click "Log QSO" in the navigation
2. Fill in the form: callsign of the station worked, band (e.g. 20m), mode (e.g. SSB), RST sent/received
3. Date and time default to now (UTC)
4. Click Submit

**## Step 4: Log a QSO via the API**

```bash
curl -X POST http://localhost:8000/api/qsos/ \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "call": "DL1ABC",
    "band": "20m",
    "mode": "SSB",
    "qso_date": "20240415",
    "time_on": "1430",
    "rst_sent": "59",
    "rst_rcvd": "59"
  }'
```
Explain: Date format is YYYYMMDD, time is HHMM or HHMMSS (UTC). OPERATOR is auto-stamped from your JWT -- you do not send it. Returns 201 with the created QSO object.

Note about duplicates: If a QSO with the same CALL+BAND+MODE+operator within +/-2 minutes already exists, you get 409 Conflict. To force creation anyway, add `?force=true` to the URL.

**## Step 5: Import QSOs from an ADIF File**

```bash
curl -X POST http://localhost:8000/api/adif/import \
  -H "Authorization: Bearer $TOKEN" \
  -F "file=@mylog.adi"
```
Explain: Returns a report showing created count, duplicate count, and any per-record errors. Duplicates are detected the same way as manual entry. ADIF import does NOT auto-stamp OPERATOR or STATION_CALLSIGN -- it preserves the values from the file (historical records).

**## Step 6: Export Your Log as ADIF**

```bash
curl http://localhost:8000/api/adif/export \
  -H "Authorization: Bearer $TOKEN" \
  -o mylog.adi
```
Explain: Exports only YOUR QSOs (operator-isolated). Output is ADIF 3.1.4 format. Soft-deleted QSOs are excluded.

**## Step 7: Watch the Station Feed**

Explain: The station feed is a live Server-Sent Events (SSE) stream showing QSOs as they are logged by any operator.

- **Browser:** Navigate to the log page -- live updates appear automatically
- **API:** The SSE endpoint at `/feed/station` requires cookie authentication (not Bearer token), because the browser's EventSource API cannot send custom headers. In the browser this works automatically. From curl:
  ```bash
  curl -N --cookie "session=..." http://localhost:8000/feed/station
  ```
  (The cookie is set during browser login.)

Explain why cookie auth: EventSource is a browser API that doesn't support custom headers. The SSE endpoint reads the JWT from an HttpOnly cookie instead.
